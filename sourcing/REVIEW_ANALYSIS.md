# 리뷰 전수 분석 방법론 (소싱 표준) v2

> 경쟁사 리뷰에서 **소비자가 왜/언제 사는지(수요) + 무엇이 불만인지(페인포인트)**를
> 정확히 뽑아 소싱·상품기획·상세페이지에 쓰는 표준 절차.
> ⚠️ **키워드 카운트 금지** ("운동"→"운동화" 오매칭). 반드시 **전수 정독 태깅**.

## 핵심 4원칙 (지난 실패에서 확립)
1. **추측 → 집계**: LLM은 분류만, 최종 수치는 코드가 카운트
2. **근거 원문 인용 의무**: 모든 결론에 실제 리뷰 원문 2~3개 첨부
3. **커버리지 명시**: "몇 건 중 몇 건 읽었는지" 항상 밝힘 (전수/샘플)
4. **추출 파이프라인화**: 추출→분할→서브에이전트 전수태깅→코드집계

## 5단계 파이프라인
1. **추출**: `review_pipeline.py extract` — Firestore/파일 리뷰 → 텍스트(15자+) 필터 → N개 chunk
2. **유형 분류**: 상품(productName)을 유형으로 묶음 (예: 족저근막형/쿠션형/키높이형)
3. **전수 태깅**: chunk별 서브에이전트(sonnet) 1명씩 → 한 건씩 정독 태깅 → `tagged_N.json` 저장
4. **집계**: `review_pipeline.py aggregate` — 코드로 축별·유형별 카운트
5. **보고**: 수치 + 원문인용 + 커버리지 + 한계

## 분석 축
**고정축** (모든 제품):
- `usage`(언제/왜 쓰나) — **배타적 우선순위**로 태깅 (아래 룰)
- `needs`(핵심니즈, 복수) — 제품별 정의
- `pain`(불만, 복수) — **1~3★ 부정리뷰 전담**. 차별화 금광
- `buyer`(성별/연령/선물여부), `repeat`(재구매)

**usage 배타 우선순위 룰** (위에서 해당되면 그것, '일상'은 최후):
1. 통증치료(질환·통증 목적) > 2. 서서일(장시간 기립) > 3. 걷기 > 4. 운동(실제 스포츠, "운동화"만 제외) > 5. 선물 > 6. 일상
- ⚠️ '일상'은 너무 넓으므로 **반드시 세분**: 신발보완 / 발피로경감 / 착화감(순수일상)
- 품질불만만 있고 사용상황 불명 → usage 미부여(pain 축으로)

**가변축** (제품 따라): 깔창=shoe/body, 탈취제=장소/대상/향 …

## ⚠️ 반드시 점검할 함정 (깔창에서 발견)
1. **모수 편향**: 베스트셀러 1개가 모수의 60% 먹을 수 있음 (닥터메모리 4,719/15,946)
   → **상품 유형별 세그먼트 분리** 집계 필수. 단 전체값=판매량비례=시장수요 반영(왜곡 아님)
2. **"일상" 과대**: 두루뭉술 → 세분(신발보완/발피로/착화감)
3. **부정리뷰 = 페인포인트**: 절대 버리지 말 것. "배송불량"은 usage엔 X지만 pain엔 핵심
4. **텍스트 커버리지**: 별점만 누른 리뷰는 분석 불가 → "글 남긴 N명 기준"임을 명시
   (깔창: 15,946건 중 텍스트 6,459건만. 부정 텍스트는 210건뿐 → 표본보강 필요시 윙 추가수집)
5. **표본 대표성(상품 수)**: 스캔이 상위 N개 판매자만 가져오면 **중·하위권 셀러 리뷰가 누락**됨.
   상위권=대형셀러 편향. 상품수를 40~100종으로 확대하면 틈새 수요·불만까지 반영됨.
   → "상위 N종 기준"임을 명시하고, 중요 의사결정 전 상품수 확대 스캔 권장.
   (깔창: 상위 24종 기준 — 향후 확대 필요)
